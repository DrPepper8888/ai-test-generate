## 1. 项目概述

### 1.1 P2 目标

结合两篇文章的核心思想，实现以下目标：

| 目标 | 说明 |
|------|------|
| **G1** | 新增"测试点拆解"独立模式 |
| **G2** | 重构记忆系统，支持规则半透明展示 |
| **G3** | 建立可持续积累的规则库 |
| **G4** | 提升生成质量，降低用户纠错成本 |

### 1.2 核心概念：Harness Engineering

**定义**：每次发现 Agent 犯错，就花时间工程化一个解决方案，让它永远不会再犯同样的错误。

**核心思想**：
- 错误沉淀到环境里，不留在人脑子里
- 规则持续进化，越用越准
- AI 是"靠谱的陪练"，不是"偷偷埋坑的代写"

## 3. 技术架构

### 3.1 新增模块

```
src/harness/
├── __init__.py           # 模块初始化
├── models.py             # 规则数据模型
├── memory_store.py       # 规则存储管理
├── rule_extractor.py     # 规则提取器
└── rule_injector.py      # 规则注入器
```

### 3.2 数据模型

```python
class RuleType(Enum):
    FIELD_FORMAT = "field_format"        # 字段格式规范
    SCENARIO_COVERAGE = "scenario_coverage"  # 场景覆盖
    BUSINESS_LOGIC = "business_logic"     # 业务逻辑
    BOUNDARY_VALUE = "boundary_value"    # 边界值
    RISK_ASSESSMENT = "risk_assessment"  # 风险评估
    GENERAL = "general"                  # 通用规则

class RuleLevel(Enum):
    GLOBAL = "global"   # 全局通用
    PROJECT = "project" # 项目级
    USER = "user"       # 用户级

@dataclass
class Rule:
    rule_id: str
    rule_text: str
    type: RuleType
    level: RuleLevel = RuleLevel.GLOBAL
    tags: List[str] = []
    source: str = ""
    source_detail: str = ""
    use_count: int = 0
    effective_count: int = 0
    ineffective_count: int = 0
    created_at: str = ""
    last_used_at: Optional[str] = None
    is_deprecated: bool = False
```

### 3.3 规则进化流程

```
用户反馈（评价差）
    ↓
RuleExtractor 提取规则 ← LLM 分析反馈
    ↓
MemoryStore 保存规则
    ↓
RuleInjector 注入到 system_prompt
    ↓
下次生成自动规避同类错误
```

```markdown
# 测试规则模板

# 测试用例必须包含的字段
1. ID：用例编号（格式：TC_XXX）
2. 标题：用例名称
3. 前置条件：必要的初始状态
4. 测试步骤：清晰的执行步骤

## 10. LLM 配置优化

### 10.1 max_tokens 设置建议

根据用途设置合适的 max_tokens：

| 用途 | 推荐值 | 说明 |
|------|--------|------|
| 简单回复 | 1024-2048 | 够用 |
| **测试用例生成** | **4096-6144** | 推荐，已优化 |
| 复杂分析 | 8192+ | 需要更多上下文 |

**当前配置**：`max_tokens: 6144`

修改方式：编辑 `config.json`
```json
{
  "llm": {
    "max_tokens": 6144  // 建议 4096-8192
  }
}
```

### 10.2 思考过程自动过滤

已实现自动过滤模型思考过程，节省 token：

```python
# 过滤前
"<think>\n分析需求...\n</think>\n好的，我来生成..."

# 过滤后
"好的，我来生成..."
```

支持的思考标签：
- `<think>...</think>` (DeepSeek)
- `<thinking>...</thinking>` (通用)
- `<reasoning>...</reasoning>` (通用)

**效果**：预计节省 20-50% 的输出 token

## 12. 请求队列功能（多人排队）

### 12.1 功能说明

解决多人同时请求时大模型并发堵塞的问题：

```
用户A 请求 → ─┐
                   ├─→ 排队中... → A处理中 → B处理中
用户B 请求 → ─┘

同时只有 1 个请求在调用大模型，避免并发堵塞
```

### 12.2 使用方式

1. 选择模式：**🔒 队列模式**
2. 点击生成
3. 如果有其他用户在排队，会显示：
   - 🔒 排队中（当前第 X 位，预计等待 Y 秒）
4. 等待轮到后自动执行

### 12.3 API 接口

| API | 方法 | 说明 |
|-----|------|------|
| `/api/queue/status` | GET | 获取队列状态 |
| `/api/queue/join` | POST | 加入队列 |
| `/api/queue/check` | POST | 检查排队状态 |

### 12.4 队列状态响应

```json
{
  "success": true,
  "total": 3,           // 总请求数
  "processing": 1,       // 处理中
  "waiting": 2,          // 排队中
  "has_queue": true,      // 是否有排队
  "estimated_wait_time": 30  // 预计等待秒数
}
```

### 12.5 实现文件

| 文件 | 说明 |
|------|------|
| `src/api/request_queue.py` | 请求队列管理 |
| `src/api/queued_llm_client.py` | 队列化 LLM 客户端 |
| `web/app.py` | 队列 API |
| `web/templates/index.html` | 队列模式 UI |

### 12.6 工作原理

```
1. 请求到来 → 加入队列尾部
2. 轮询检查是否轮到（队首）
3. 轮到 → 标记为"处理中" → 调用 LLM
4. 完成 → 标记为"完成" → 通知下一位
5. 使用文件锁确保线程安全
```

### 12.7 优势

- **防堵塞**：同一时间只调用一次 LLM
- **自动排队**：用户无感知
- **状态透明**：实时显示排队位置
- **零依赖**：纯 Python 标准库

## 13. 任务驱动队列（已重构）

### 13.1 设计变更

| 之前（用户驱动） | 现在（任务驱动） |
|-----------------|-----------------|
| 需要识别用户 IP | 不依赖 IP，统一排队 |
| 用户标识不准确 | 每个任务有唯一 ID |
| IP 共享时混乱 | 共享 IP 也能正确排队 |

### 13.2 核心设计

```
┌─────────────────────────────────────────┐
│           TaskQueue（单例）             │
│  ┌─────────────────────────────────┐  │
│  │  队列: [task1, task2, task3...]  │  │
│  │                                 │  │
│  │  task1: status=processing      │  │
│  │  task2: status=waiting, pos=2   │  │
│  │  task3: status=waiting, pos=3   │  │
│  └─────────────────────────────────┘  │
└─────────────────────────────────────────┘
```

### 13.3 API 更新

| API | 说明 |
|-----|------|
| `GET /api/queue/status` | 获取队列状态 |
| `POST /api/queue/add` | 添加工任务 |
| `POST /api/queue/check` | 检查任务状态 |

### 13.4 使用方式

1. 选择 **🔒 队列模式**
2. 点击生成
3. 系统自动：
   - 添加工任务
   - 排队等待
   - 串行执行
   - 自动释放

### 13.5 优势

- **无需用户标识**：不管是谁发起的请求，统一排队
- **共享 IP 支持**：多人共用一个 IP 也能正确排队
- **单例模式**：全局共享队列
- **线程安全**：文件锁保护
